## [2026-04-14]

### 🕳现象描述

构建流程在 **Ninja 后端执行阶段失败**。CMake 生成 `build.ninja` 成功，但在实际调用编译器（GCC/MinGW）执行编译或链接子任务时，某个命令返回状态码1，导致 Ninja 触发 `subcommand failed` 中断。该错误未提供具体的源码报错行号，属于**构建系统级阻断**，而非语法错误。

```
ninja: build stopped: subcommand failed.
18:39:45: 进程"D:\Qt\Tools\CMake_64\bin\cmake.exe"退出，退出代码 1 。
18:39:45: Error while building/deploying project 41401 (kit: Desktop Qt 5.15.2 MinGW 32-bit)
18:39:45: When executing step "构建"
```

### 👁问题定位

```
# CMakeList.txt 
# 让 CMake 自动发现并注册测试用例（需要 GoogleTest 包）
include(GoogleTest)
gtest_discover_tests(unit_tests)
```

### 🧠根因分析


1. **机制依赖**  
   `gtest_discover_tests` 的工作方式并非静态注册，而是在构建时**动态执行**测试程序以收集内部用例列表。该步骤依赖于被测程序能在 CMake 的子进程环境中**干净启动并正常退出**。

2. **环境冲突**  
   当前使用的 `Desktop Qt 5.15.2 MinGW 32-bit` 套件采用 **DWARF 异常模型**。

   当 CMake 调用 Ninja 执行 `POST_BUILD` 发现步骤时，系统 `PATH` 环境变量未正确包含 Qt 及 MinGW 运行时库路径，导致 `unit_tests.exe` 启动瞬间因找不到 `libgcc_s_dw2-1.dll` 或 `libstdc++-6.dll` 而返回错误码 `1`，触发 `subcommand failed`。

> **结论**：MinGW 32-bit 工具链与 `gtest_discover_tests` 的动态发现机制存在**运行时环境兼容性冲突**

### 🔥实际解决路径

1. **降级为静态注册** 
   使用 `add_test` 直接声明测试目标，**切除**对动态发现机制的依赖

   ```
   # 让 CMake 自动发现并注册测试用例（需要 GoogleTest 包）
   # include(GoogleTest)
   # gtest_discover_tests(unit_tests)
   
   # 手动将测试程序注册到 CTest
   add_test(NAME unit_tests COMMAND unit_tests)
   ```

2. **升级工具链**   

   改用 **MinGW 64-bit（SEH 异常模型）**，其运行时库加载机制与 Windows 子系统兼容性更好，修复了子进程启动环境。
